## 📦 Import 자동 정렬 (✅ 린트 설정)

### 라이브러리: `eslint-plugin-simple-import-sort`

**선택 이유:**

- 설정이 간단하고 직관적

- ESLint `--fix` 옵션으로 자동 수정

### Import 정렬 순서

```typescript
// 1. React 관련
```

## 🏗️ FSD 아키텍처 규칙 (✅ 린트 설정)

### 레이어 구조 및 의존성 규칙

```
app      → widgets, features, entities, shared (모두 가능)
widgets  → features, entities, shared
features → entities, shared
entities → shared
shared   → (다른 레이어 import 불가)
```

**ESLint 규칙:**

```javascript
'import/no-restricted-paths': [
  'error',
  {
    zones: [
      // shared는 다른 레이어 import 불가
      { target: './src/shared/**/*', from: './src/entities/**/*' },
      { target: './src/shared/**/*', from: './src/features/**/*' },
      { target: './src/shared/**/*', from: './src/widgets/**/*' },

      // entities는 features 이상 레이어 import 불가
      { target: './src/entities/**/*', from: './src/features/**/*' },
      { target: './src/entities/**/*', from: './src/widgets/**/*' },

      // features는 widgets 이상 레이어 import 불가
      { target: './src/features/**/*', from: './src/widgets/**/*' }
    ]
  }
]
```

**예시:**

```typescript
// ❌ 에러 발생 (린트가 감지)
// src/features/auth/login-form.tsx
import { Header } from "@/widgets/header";
// Error: features 레이어는 widgets를 import할 수 없습니다.

// ✅ 정상 작동
// src/features/auth/login-form.tsx
import { Button } from "@/shared/ui/button";
import { User } from "@/entities/user";
```

## ⚛️ React 컨벤션

### 1. 화살표 함수 사용 (✅ 린트 설정)

**규칙**: 모든 React 컴포넌트는 **화살표 함수**로 작성

```typescript
// ✅ 올바른 방식
export const LoginPage = () => {
  return <div>Login</div>;
};

// ❌ 잘못된 방식
export function LoginPage() {
  return <div>Login</div>;
}
```

**왜 화살표 함수를 사용하나?**

1. **일관성**: 프로젝트 전체에서 동일한 스타일 유지
2. **this 바인딩 문제 해결**: React에서 발생할 수 있는 `this` 바인딩 문제 방지
3. **Named Export와 호환성**: `export const`와 자연스럽게 결합

**ESLint 규칙:**

```javascript
'react/function-component-definition': ['error', {
  namedComponents: 'arrow-function',
  unnamedComponents: 'arrow-function'
}]
```

---

### 2. Named Export 사용 (✅ 린트 설정)

**규칙**: 컴포넌트는 **Named Export** 사용 (Default Export 금지)

```typescript
// ✅ Named Export (권장)
export const LoginButton = () => {};
export const LoginForm = () => {};

// ❌ Default Export (지양)
export default function LoginButton() {}
```

**왜 Named Export를 사용하나?**

1. **리팩토링 안정성**: 파일명 변경 시 import 자동 업데이트
2. **명시성**: import 시 정확한 이름 사용, 오타 방지
3. **IDE 지원**: 자동완성, Go to Definition 등 더 나은 지원
4. **트리 쉐이킹**: 번들러가 미사용 export 제거 가능

**예시:**

```typescript
// Named Export: 리팩토링 시 안전
// user-profile.tsx
export const UserProfile = () => {};

// other-file.tsx
import { UserProfile } from "./user-profile"; // ✅ 파일명 변경 시 자동 업데이트

// Default Export: 리팩토링 시 위험
// user-profile.tsx
export default function UserProfile() {}

// other-file.tsx
import Profile from "./user-profile"; // ⚠️ 이름이 달라도 에러 없음
import AnythingHere from "./user-profile"; // ⚠️ 혼란 유발
```

**ESLint 규칙:**

```javascript
// 기본적으로 error, Next.js 특수 파일 및 설정 파일만 오버라이드로 허용
'import/no-default-export': 'error'

// 오버라이드: Next.js 특수 파일은 Default Export 허용
{
  files: [
    'src/app/**/page.tsx',
    'src/app/**/layout.tsx',
    'src/app/**/error.tsx',
    'src/app/**/loading.tsx',
    'src/app/**/not-found.tsx',
    'src/app/**/template.tsx',
    'src/app/**/default.tsx',
    'src/pages/**/*.tsx',
    'src/pages/api/**/*.ts',
    '*.config.{js,ts,mjs,cjs}',
  ],
  rules: {
    'import/no-default-export': 'off',
  },
}
```

**참고**: Next.js의 `app/page.tsx`, `app/layout.tsx` 등과 설정 파일만 Default Export 허용

### 2. 'use client' 최소화

**규칙**: `'use client'`는 필요한 곳에만 사용


### 2. Type vs Interface (✅ 린트 설정)

**규칙**: 일관성을 위해 **type 우선 사용**

```typescript
// ✅ type 사용 (권장)
type ButtonProps = {
  label: string;
  onClick: () => void;
};

// ❌ interface 사용 (지양)
interface ButtonProps {
  label: string;
  onClick: () => void;
}
```

**왜 type을 우선 사용하나?**

1. **일관성 (Consistency)**

   ```typescript
   // type은 모든 경우에 사용 가능
   type Name = string;  // ✅ Primitive alias
   type Point = { x: number; y: number };  // ✅ Object
   type ID = string | number;  // ✅ Union

   // interface는 객체 형태만 가능
   interface Point { x: number; y: number }  // ✅
   interface ID = string | number;  // ❌ 불가능
   ```

2. **합성(Composition) 유연성**

   ```typescript
   // type: Union, Intersection 자유롭게
   type Loading = { status: 'loading' };
   type Success = { status: 'success'; data: string };
   type Error = { status: 'error'; error: string };
   type State = Loading | Success | Error;  // ✅ Union

   // interface: Union 불가능
   interface State = Loading | Success | Error;  // ❌
   ```

3. **실무 트렌드**
   - Modern React/Next.js 프로젝트: **type 우선** (Vercel, Airbnb 등)
   - 라이브러리 개발: **interface 우선** (확장성 필요)

**언제 interface를 사용하나?**

확장이 명확히 필요한 경우 (주로 라이브러리 개발):

```typescript
// ✅ 계층 구조가 명확한 경우
interface BaseEntity {
  id: string;
  createdAt: Date;
}

interface User extends BaseEntity {
  name: string;
}

interface Admin extends User {
  permissions: string[];
}

// ✅ 외부에서 확장 가능해야 하는 경우 (Declaration Merging)
// lib.d.ts
interface Window {
  myCustomProperty: string;
}
```

**ESLint 규칙:**

```javascript
'@typescript-eslint/consistent-type-definitions': ['warn', 'type']
```

**결론**: 일반적인 애플리케이션 코드는 **type**, 확장이 필수인 경우만 **interface** 사용

### 3. `@apply` 최소화

**규칙**: Tailwind 장점을 살리기 위해 `@apply` 사용 최소화

**왜 `@apply`를 피해야 하나?**

1. **번들 크기 증가**: 같은 스타일이 여러 클래스로 중복 생성
2. **PurgeCSS 효과 감소**: 사용하지 않는 CSS가 제거되지 않음
3. **Tailwind 장점 상실**: 유틸리티 클래스의 재사용성 감소

**예시:**

```scss
// ❌ 나쁜 방식: @apply 남발
.button-primary {
  @apply flex items-center px-4 py-2 bg-blue-500 rounded-md;
}

.button-secondary {
  @apply flex items-center px-4 py-2 bg-gray-200 rounded-md;
}
// 문제: 같은 스타일(flex, items-center, px-4, py-2, rounded-md)이 2번 중복 생성
```

```tsx
// ✅ 좋은 방식: 컴포넌트로 추상화
type ButtonProps = {
  variant?: 'primary' | 'secondary';
  children: React.ReactNode;
};

export const Button = ({ variant = 'primary', children }: ButtonProps) => {
  const variants = {
    primary: 'bg-blue-500 text-white',
    secondary: 'bg-gray-200 text-gray-900',
  };

  return (
    <button className={`flex items-center px-4 py-2 rounded-md ${variants[variant]}`}>
      {children}
    </button>
  );
};

// 사용: 유틸리티 클래스 재사용, 중복 없음
<Button variant="primary">Submit</Button>
<Button variant="secondary">Cancel</Button>
```

**@apply를 써도 되는 경우 (매우 제한적):**

```scss
// ✅ 예외 1: 기본 HTML 태그 스타일링
@layer base {
  h1 {
    @apply text-4xl font-bold mb-4;
  }

  h2 {
    @apply text-3xl font-bold mb-3;
  }
}

// ✅ 예외 2: 서드파티 라이브러리 커스터마이징
.react-datepicker {
  @apply border border-gray-300 rounded-lg shadow-lg;
}
```

### 4. SCSS Module 사용 시

**사용 시기:** 복잡한 애니메이션, 의사 요소, 복잡한 선택자가 필요한 경우

```scss
// ComplexAnimation.module.scss
.animatedCard {
  position: relative;
  overflow: hidden;

  // 복잡한 의사 요소 애니메이션
  &::before {
    content: "";
    position: absolute;
    top: -50%;
    left: -50%;
    width: 200%;
    height: 200%;
    background: linear-gradient(
      45deg,
      transparent,
      rgba(255, 255, 255, 0.1),

    0% {
      transform: translateX(-100%) translateY(-100%) rotate(45deg);
    }
    100% {
      transform: translateX(100%) translateY(100%) rotate(45deg);
    }
  }
}
```

```tsx
// ✅ Sass Module (애니메이션) + Tailwind (레이아웃) 조합

export const AnimatedCard = () => {

    </div>

};
```

- 파일명: `component.module.scss`
- 클래스명: camelCase (예: `.animatedCard`, `.loadingSpinner`)

## 📂 파일명 컨벤션

### 기본 원칙: 파일 유형에 따라 다르게 적용

**파일의 역할**에 따라 적절한 네이밍 컨벤션을 사용.

---

### 1. React 컴포넌트: **PascalCase** ✅

**규칙**: 컴포넌트 파일은 **PascalCase** 사용 (Airbnb 스타일 가이드 기준)

```
✅ LoginButton.tsx
✅ UserProfile.tsx
✅ DataTable.tsx

❌ login-button.tsx
❌ userProfile.tsx
❌ data_table.tsx
```

**왜 PascalCase를 사용하나?**

1. **컴포넌트명과 파일명 일치**: 명확성과 직관성

   ```typescript
   // LoginButton.tsx
   export const LoginButton = () => { ... }

   // 다른 파일에서
   import { LoginButton } from './LoginButton';  // 이름이 일치!
   ```

2. **Airbnb 스타일 가이드 권장**: 가장 널리 사용되는 React 스타일 가이드
3. **IDE 지원 향상**: 자동완성, Go to Definition 등에서 더 나은 경험
4. **트리 쉐이킹 최적화**: 번들러가 미사용 컴포넌트 제거 용이

---

### 2. Custom Hooks: **camelCase**

**규칙**: Hook 파일은 **camelCase** 사용 (`use` 접두사 포함)

```
✅ useGoogleLogin.ts
✅ useUserData.ts
✅ useDebounce.ts

❌ UseGoogleLogin.ts
❌ use-google-login.ts
```

**이유**: React Hook 네이밍 컨벤션과 일치

```typescript
// useGoogleLogin.ts
export const useGoogleLogin = () => { ... }

// 사용
import { useGoogleLogin } from './useGoogleLogin';
```

---

### 3. 유틸리티/함수: **camelCase**

**규칙**: 일반 함수/유틸 파일은 **camelCase** 사용

```
✅ formatDate.ts
✅ validateEmail.ts
✅ apiClient.ts

❌ FormatDate.ts
❌ format-date.ts
```

**이유**: JavaScript 변수 네이밍 컨벤션과 일치

---

### 4. Store/모델: **camelCase**

**규칙**: 상태 관리, 데이터 모델 파일은 **camelCase** 사용

```
✅ authStore.ts
✅ userModel.ts
✅ todoSlice.ts

❌ AuthStore.ts
❌ auth-store.ts
```

---

### 5. Next.js 특수 파일: **kebab-case (강제)**

**규칙**: Next.js App Router의 특수 파일은 **kebab-case** (Next.js 강제 규칙)

```
✅ page.tsx
✅ layout.tsx
✅ loading.tsx
✅ error.tsx
✅ not-found.tsx
✅ route.ts
```

**폴더명도 kebab-case** (URL 경로가 됨)

---

### 6. 타입 정의 파일: **camelCase 또는 PascalCase**

**규칙**: 타입만 정의하는 파일은 **둘 다 가능** (일관성 유지)

```
✅ userTypes.ts  (camelCase - 선호)
✅ UserTypes.ts  (PascalCase - 허용)

✅ types.ts
✅ index.ts
```
