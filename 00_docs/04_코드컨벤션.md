## 🏗️ FSD 아키텍처 규칙 (✅ 린트 설정)

### 레이어 구조 및 의존성 규칙

```
app      → widgets, features, entities, shared (모두 가능)
widgets  → features, entities, shared
features → entities, shared
entities → shared
shared   → (다른 레이어 import 불가)
```

**ESLint 규칙:**

```javascript
'import/no-restricted-paths': [
  'error',
  {
    zones: [
      // shared는 다른 레이어 import 불가
      { target: './src/shared/**/*', from: './src/entities/**/*' },
      { target: './src/shared/**/*', from: './src/features/**/*' },
      { target: './src/shared/**/*', from: './src/widgets/**/*' },

      // entities는 features 이상 레이어 import 불가
      { target: './src/entities/**/*', from: './src/features/**/*' },
      { target: './src/entities/**/*', from: './src/widgets/**/*' },

      // features는 widgets 이상 레이어 import 불가
      { target: './src/features/**/*', from: './src/widgets/**/*' }
    ]
  }
]
```

**예시:**

```typescript
// ❌ 에러 발생 (린트가 감지)
```

## ⚛️ React 컨벤션

### 1. 화살표 함수 사용 (✅ 린트 설정)

**규칙**: 모든 React 컴포넌트는 **화살표 함수**로 작성

```typescript
// ✅ 올바른 방식
export const LoginPage = () => {
  return <div>Login</div>;
};

// ❌ 잘못된 방식
export function LoginPage() {
  return <div>Login</div>;
}
```

**왜 화살표 함수를 사용하나?**

1. **일관성**: 프로젝트 전체에서 동일한 스타일 유지
2. **this 바인딩 문제 해결**: React에서 발생할 수 있는 `this` 바인딩 문제 방지
3. **Named Export와 호환성**: `export const`와 자연스럽게 결합

**ESLint 규칙:**

```javascript
'react/function-component-definition': ['error', {
  namedComponents: 'arrow-function',
  unnamedComponents: 'arrow-function'
}]
```

### 2. Named Export 사용 (✅ 린트 설정)

**규칙**: 컴포넌트는 **Named Export** 사용 (Default Export 금지)

```typescript
// ✅ Named Export (권장)
export const LoginButton = () => {};
export const LoginForm = () => {};

// ❌ Default Export (지양)
export default function LoginButton() {}
```

**왜 Named Export를 사용하나?**

1. **리팩토링 안정성**: 파일명 변경 시 import 자동 업데이트
2. **명시성**: import 시 정확한 이름 사용, 오타 방지
3. **IDE 지원**: 자동완성, Go to Definition 등 더 나은 지원
4. **트리 쉐이킹**: 번들러가 미사용 export 제거 가능

**예시:**

```typescript
// Named Export: 리팩토링 시 안전
// user-profile.tsx
export const UserProfile = () => {};

// other-file.tsx
import { UserProfile } from "./user-profile"; // ✅ 파일명 변경 시 자동 업데이트

// Default Export: 리팩토링 시 위험
// user-profile.tsx
export default function UserProfile() {}

// other-file.tsx
import Profile from "./user-profile"; // ⚠️ 이름이 달라도 에러 없음
import AnythingHere from "./user-profile"; // ⚠️ 혼란 유발
```

**ESLint 규칙:**

```javascript
// 기본적으로 error, Next.js 특수 파일 및 설정 파일만 오버라이드로 허용
'import/no-default-export': 'error'

// 오버라이드: Next.js 특수 파일은 Default Export 허용
{
  files: [
    'src/app/**/page.tsx',
    'src/app/**/layout.tsx',
    'src/app/**/error.tsx',
    'src/app/**/loading.tsx',
    'src/app/**/not-found.tsx',
    'src/app/**/template.tsx',
    'src/app/**/default.tsx',
    'src/pages/**/*.tsx',
    'src/pages/api/**/*.ts',
    '*.config.{js,ts,mjs,cjs}',
  ],
  rules: {
    'import/no-default-export': 'off',
  },
}
```

**참고**: Next.js의 `app/page.tsx`, `app/layout.tsx` 등과 설정 파일만 Default Export 허용

### 1. API Route 구조

**폴더 구조:**

```
app/api/
├── books/
│   ├── route.ts              # GET /api/books (목록), POST /api/books (생성)
│   └── [id]/
│       └── route.ts          # GET /api/books/:id (상세)
│                             # PATCH /api/books/:id (수정)
│                             # DELETE /api/books/:id (삭제)
├── comments/
│   └── route.ts
└── likes/
    └── route.ts
```

**HTTP 메서드 매핑:**

| 작업 | HTTP 메서드 | 엔드포인트 | 설명 |
|------|------------|-----------|------|
| 목록 조회 | GET | `/api/books` | 쿼리 파라미터로 필터링 |
| 상세 조회 | GET | `/api/books/:id` | 특정 책 정보 |
| 생성 | POST | `/api/books` | 새 책 추가 |
| 수정 | PATCH | `/api/books/:id` | 부분 수정 |
| 삭제 | DELETE | `/api/books/:id` | 책 삭제 |

**예시:**

```typescript
// app/api/books/route.ts
import { NextRequest, NextResponse } from 'next/server';
import { createClient } from '@/shared/config/supabase/server';

// GET /api/books?status=reading
export async function GET(request: NextRequest) {
  const supabase = await createClient();
  const searchParams = request.nextUrl.searchParams;
  const status = searchParams.get('status');

  let query = supabase.from('books').select('*').order('created_at', {
    ascending: false,
  });

  if (status) {
    query = query.eq('status', status);
  }

  const { data, error } = await query;

  if (error) {
    return NextResponse.json({ error: error.message }, { status: 500 });
  }

  return NextResponse.json(data);
}

// POST /api/books
export async function POST(request: NextRequest) {
  const supabase = await createClient();
  const body = await request.json();

  const {
    data: { user },
  } = await supabase.auth.getUser();

  if (!user) {
    return NextResponse.json({ error: '로그인이 필요합니다.' }, { status: 401 });
  }

  const { data, error } = await supabase
    .from('books')
    .insert({
      ...body,
      user_id: user.id,
    })
    .select()
    .single();

  if (error) {
    return NextResponse.json({ error: error.message }, { status: 500 });
  }

  return NextResponse.json(data, { status: 201 });
}
```

```typescript
// app/api/books/[id]/route.ts
import { NextRequest, NextResponse } from 'next/server';
import { createClient } from '@/shared/config/supabase/server';

type Params = {
  params: Promise<{ id: string }>;
};

// GET /api/books/:id
export async function GET(request: NextRequest, { params }: Params) {
  const supabase = await createClient();
  const { id } = await params;

  const { data, error } = await supabase
    .from('books')
    .select('*')
    .eq('id', id)
    .single();

  if (error) {
    return NextResponse.json({ error: error.message }, { status: 404 });
  }

  return NextResponse.json(data);
}

// PATCH /api/books/:id
export async function PATCH(request: NextRequest, { params }: Params) {
  const supabase = await createClient();
  const { id } = await params;
  const body = await request.json();

  const { data, error } = await supabase
    .from('books')
    .update({
      ...body,
      updated_at: new Date().toISOString(),
    })
    .eq('id', id)
    .select()
    .single();

  if (error) {
    return NextResponse.json({ error: error.message }, { status: 500 });
  }

  return NextResponse.json(data);
}

// DELETE /api/books/:id
export async function DELETE(request: NextRequest, { params }: Params) {
  const supabase = await createClient();
  const { id } = await params;

  const { error } = await supabase.from('books').delete().eq('id', id);

  if (error) {
    return NextResponse.json({ error: error.message }, { status: 500 });
  }

  return NextResponse.json({ message: '삭제되었습니다.' });
}
```

### 2. axios 클라이언트 설정

**파일 위치:** `src/shared/api/client.ts`

**역할:**

- axios 인스턴스 생성
- 공통 설정 (baseURL, timeout)
- 인터셉터로 인증 토큰 자동 추가
- 에러 핸들링 통합

**예시:**

```typescript
// src/shared/api/client.ts
import axios from 'axios';

export const apiClient = axios.create({
  baseURL: '/api',
  timeout: 10000,
  headers: {
    'Content-Type': 'application/json',
  },
});

// 요청 인터셉터 (인증 토큰 자동 추가 등)
apiClient.interceptors.request.use(
  (config) => {
    // 향후 인증 토큰 추가 가능
    return config;
  },
  (error) => {
    return Promise.reject(error);
  }
);

// 응답 인터셉터 (에러 핸들링)
apiClient.interceptors.response.use(
  (response) => response,
  (error) => {
    if (error.response?.status === 401) {
      // 인증 에러 처리
      console.error('인증이 필요합니다.');
    }
    return Promise.reject(error);
  }
);
```

```typescript
import { useState } from 'react';
import { bookApi } from '@/entities/book/api/bookApi';

export const useBookCRUD = () => {
  const [isLoading, setIsLoading] = useState(false);
  const [error, setError] = useState<string | null>(null);

  const getMyBooks = async (status?: string) => {
    setIsLoading(true);
    try {
      const data = await bookApi.getBooks(status);
      return data;
    } catch (err) {
      setError(err instanceof Error ? err.message : '조회 실패');
      return [];
    } finally {
      setIsLoading(false);
    }
  };

  return { isLoading, error, getMyBooks };
```

### 2. Type vs Interface (✅ 린트 설정)

**규칙**: 일관성을 위해 **type 우선 사용**

```typescript
// ✅ type 사용 (권장)
type ButtonProps = {
  label: string;
  onClick: () => void;
};

// ❌ interface 사용 (지양)
interface ButtonProps {
  label: string;
  onClick: () => void;
}
```

**왜 type을 우선 사용하나?**

1. **일관성 (Consistency)**

   ```typescript
   // type은 모든 경우에 사용 가능
   type Name = string;  // ✅ Primitive alias
   type Point = { x: number; y: number };  // ✅ Object
   type ID = string | number;  // ✅ Union

   // interface는 객체 형태만 가능
   interface Point { x: number; y: number }  // ✅
   interface ID = string | number;  // ❌ 불가능
   ```

2. **합성(Composition) 유연성**

   ```typescript
   // type: Union, Intersection 자유롭게
   type Loading = { status: 'loading' };
   type Success = { status: 'success'; data: string };
   type Error = { status: 'error'; error: string };
   type State = Loading | Success | Error;  // ✅ Union

   // interface: Union 불가능
   interface State = Loading | Success | Error;  // ❌
   ```

3. **실무 트렌드**
   - Modern React/Next.js 프로젝트: **type 우선** (Vercel, Airbnb 등)
   - 라이브러리 개발: **interface 우선** (확장성 필요)

**언제 interface를 사용하나?**

확장이 명확히 필요한 경우 (주로 라이브러리 개발):

```typescript
// ✅ 계층 구조가 명확한 경우
interface BaseEntity {
  id: string;
  createdAt: Date;
}

interface User extends BaseEntity {
  name: string;
}

interface Admin extends User {
  permissions: string[];
}

// ✅ 외부에서 확장 가능해야 하는 경우 (Declaration Merging)
// lib.d.ts
interface Window {
  myCustomProperty: string;
}
```

**ESLint 규칙:**

```javascript
'@typescript-eslint/consistent-type-definitions': ['warn', 'type']
```

**결론**: 일반적인 애플리케이션 코드는 **type**, 확장이 필수인 경우만 **interface** 사용

---

### 3. any 금지 (✅ 린트 설정)

```typescript
// ❌ any 사용 금지
const data: any = fetchData();

// ✅ 타입 정의 또는 unknown 사용
const data: UserData = fetchData();

```

**ESLint 규칙:**

```javascript

```

### 3. `@apply` 최소화

**규칙**: Tailwind 장점을 살리기 위해 `@apply` 사용 최소화

**왜 `@apply`를 피해야 하나?**

1. **번들 크기 증가**: 같은 스타일이 여러 클래스로 중복 생성
2. **PurgeCSS 효과 감소**: 사용하지 않는 CSS가 제거되지 않음
3. **Tailwind 장점 상실**: 유틸리티 클래스의 재사용성 감소

**예시:**

```scss
// ❌ 나쁜 방식: @apply 남발
.button-primary {
  @apply flex items-center px-4 py-2 bg-blue-500 rounded-md;
}

.button-secondary {
  @apply flex items-center px-4 py-2 bg-gray-200 rounded-md;
}
// 문제: 같은 스타일(flex, items-center, px-4, py-2, rounded-md)이 2번 중복 생성
```

```tsx
// ✅ 좋은 방식: 컴포넌트로 추상화
type ButtonProps = {
  variant?: 'primary' | 'secondary';
  children: React.ReactNode;
};

export const Button = ({ variant = 'primary', children }: ButtonProps) => {
  const variants = {
    primary: 'bg-blue-500 text-white',
    secondary: 'bg-gray-200 text-gray-900',
  };

  return (
    <button className={`flex items-center px-4 py-2 rounded-md ${variants[variant]}`}>
      {children}
    </button>
  );
};

// 사용: 유틸리티 클래스 재사용, 중복 없음
<Button variant="primary">Submit</Button>
<Button variant="secondary">Cancel</Button>
```

**@apply를 써도 되는 경우 (매우 제한적):**

```scss
// ✅ 예외 1: 기본 HTML 태그 스타일링
@layer base {
  h1 {
    @apply text-4xl font-bold mb-4;
  }

  h2 {
    @apply text-3xl font-bold mb-3;
  }
}

// ✅ 예외 2: 서드파티 라이브러리 커스터마이징
.react-datepicker {
  @apply border border-gray-300 rounded-lg shadow-lg;
}
```

### 4. SCSS Module 사용 시

**사용 시기:** 복잡한 애니메이션, 의사 요소, 복잡한 선택자가 필요한 경우

```scss
// ComplexAnimation.module.scss
.animatedCard {
  position: relative;
  overflow: hidden;

  // 복잡한 의사 요소 애니메이션
  &::before {

    top: -50%;
    left: -50%;
    width: 200%;

      45deg,
```


**네이밍 규칙:**

### 1. React 컴포넌트: **PascalCase** ✅

**규칙**: 컴포넌트 파일은 **PascalCase** 사용 (Airbnb 스타일 가이드 기준)

## 🏗️ UI-비즈니스 로직 분리 원칙

### 기본 원칙: 관심사의 분리 (Separation of Concerns)

**FSD 구조에서 features 레이어는 UI와 비즈니스 로직을 명확히 분리합니다.**

```
features/
  └── [feature-name]/
      ├── api/          ← 비즈니스 로직 (Custom Hooks, API 호출)
      ├── ui/           ← UI 컴포넌트 (재사용 가능한 컴포넌트)
      └── model/        ← 상태 관리 (Zustand store, 필요시)
```

---

### 1. 비즈니스 로직: `api/` 폴더

**역할**: 데이터 처리, API 호출, 비즈니스 로직만 담당

**파일 형태**:

- Custom Hooks (`useXxx.ts`)
- API 함수 (`xxxApi.ts`)
- 서비스 함수 (`xxxService.ts`)

**예시:**

```typescript
// features/auth/api/useGoogleLogin.ts
import { createClient } from "@/shared/config/supabase/client";

export const useGoogleLogin = () => {
  const login = async () => {
    const supabase = createClient();
    const siteUrl = process.env.NEXT_PUBLIC_SITE_URL || location.origin;

    const { error } = await supabase.auth.signInWithOAuth({
      provider: "google",
      options: { redirectTo: `${siteUrl}/auth/callback` },
    });

    if (error) {
      console.error("로그인 오류:", error.message);
    }
  };

  return { login };
};
```

**특징**:

- ✅ UI 코드 없음 (JSX 없음)
- ✅ 로직만 제공 (함수 반환)
- ✅ 어디서든 재사용 가능
- ✅ 테스트 용이

---

### 2. UI 컴포넌트: `ui/` 폴더

**역할**: 사용자 인터페이스만 담당

**파일 형태**:

- React 컴포넌트 (`XxxButton.tsx`, `XxxForm.tsx`)

**예시:**

```typescript
// features/auth/ui/LoginWithGoogleButton.tsx
"use client";

import { Button } from "@/shared/ui/button";
import { Icons } from "@repo/icons";
import { useGoogleLogin } from "../api/useGoogleLogin";

export const LoginWithGoogleButton = ({
  size = "lg",
  className = "w-full",
}: LoginWithGoogleButtonProps) => {
  // api에서 로직 가져오기
  const { login } = useGoogleLogin();

  return (
    <Button onClick={login} className={className} size={size}>
      <Icons name="google" />
      Google로 로그인
    </Button>
  );
};
```

**특징**:

- ✅ UI만 담당 (JSX, 스타일)
- ✅ Props로 커스터마이징 가능
- ✅ 비즈니스 로직은 Hook에서 가져옴
- ✅ Storybook 등 UI 테스트 용이

---

### 3. 언제 분리하는가?

#### **분리가 필요한 경우 ✅**

**1. 로직을 여러 곳에서 재사용**

```typescript
// ✅ Hook으로 분리
const { login } = useGoogleLogin();

// 다양한 UI에서 사용 가능
<Button onClick={login}>로그인</Button>
<IconButton onClick={login} />
<Link onClick={login}>Google 계정으로 시작</Link>
```

**2. 로직이 복잡한 경우**

```typescript
// ✅ 복잡한 로직은 Hook으로 분리
export const useBookSearch = () => {
  const [books, setBooks] = useState([]);
  const [isLoading, setIsLoading] = useState(false);
  const [error, setError] = useState(null);

  const search = async (query: string) => {
    setIsLoading(true);
    try {
      const response = await fetch(`/api/books?q=${query}`);
      const data = await response.json();
      setBooks(data);
    } catch (err) {
      setError(err);
    } finally {
      setIsLoading(false);
    }
  };

  return { books, isLoading, error, search };
};
```

**3. 테스트가 필요한 로직**

```typescript
// ✅ 로직만 분리하면 테스트 쉬움
describe("useGoogleLogin", () => {
  it("should call signInWithOAuth", async () => {
    const { login } = useGoogleLogin();
    await login();
    expect(supabase.auth.signInWithOAuth).toHaveBeenCalled();
  });
});
```

#### **분리가 불필요한 경우 ❌**

**1. 간단한 UI 이벤트**

```typescript
// ❌ 분리 불필요: 간단한 로직
const handleClick = () => {
  console.log("clicked");
};

// 그냥 컴포넌트 안에서 처리
<Button onClick={handleClick}>클릭</Button>;
```

**2. 한 곳에서만 사용**

```typescript
// ❌ 분리 불필요: 재사용 안 함
const DashboardPage = () => {
  const handleLogout = async () => {
    await supabase.auth.signOut();
    router.push("/login");
  };

  return <Button onClick={handleLogout}>로그아웃</Button>;
};
```

**3. UI와 강하게 결합**

```typescript
// ❌ 분리 불필요: UI 상태와 밀접
const [isOpen, setIsOpen] = useState(false);
const handleToggle = () => setIsOpen(!isOpen);

<Modal isOpen={isOpen} onClose={handleToggle} />;
```

---

### 5. 핵심 원칙 요약

| 구분          | api/                   | ui/                                    |
| ------------- | ---------------------- | -------------------------------------- |
| **역할**      | 비즈니스 로직          | 사용자 인터페이스                      |
| **파일 형태** | Custom Hooks, API 함수 | React 컴포넌트                         |
| **반환값**    | 함수, 데이터, 상태     | JSX                                    |
| **의존성**    | shared만 사용          | api + shared 사용                      |
| **테스트**    | 로직 테스트 (Jest)     | UI 테스트 (Storybook, Testing Library) |
| **재사용성**  | 높음 (어디서든 호출)   | 보통 (UI 맥락 필요)                    |

**✅ 좋은 예시:**

```typescript
// api/useBookmark.ts - 로직만
export const useBookmark = () => {
  const addBookmark = async (bookId: string) => { ... };

  return { addBookmark, removeBookmark };

// ui/BookmarkButton.tsx - UI만
export const BookmarkButton = ({ bookId }: Props) => {
  const { addBookmark, removeBookmark } = useBookmark();
  return <Button onClick={() => addBookmark(bookId)}>북마크</Button>;
};
```

```typescript
// ❌ ui 안에 로직 포함
export const BookmarkButton = ({ bookId }: Props) => {
  const addBookmark = async () => {
    const supabase = createClient();
```
