> 선택한 기술과 그 이유

## Frontend

### Next.js 16 (App Router)

**선택 이유:**

- ✅ React 기반 풀스택 프레임워크
- ✅ App Router로 최신 패턴 학습
- ✅ SSR/SSG로 SEO 최적화
- ✅ Vercel 배포 최적화
- ✅ React 19 지원

**대안:**

- ❌ Vite + React: CSR만 지원
- ❌ Remix: 레퍼런스 부족

### TypeScript

**선택 이유:**

- ✅ 타입 안정성
- ✅ IDE 자동완성
- ✅ 리팩토링 용이

### Tailwind CSS v4 + shadcn/ui

**선택 이유:**

- ✅ 유틸리티 우선 CSS
- ✅ Tailwind v4: 성능 개선, PostCSS 기반
- ✅ shadcn/ui로 빠른 UI 구축
- ✅ 커스터마이징 쉬움

**대안:**

- ❌ MUI: 무겁고 커스터마이징 어려움
- ❌ Styled-components: 런타임 오버헤드

### React Query (TanStack Query)

**선택 이유:**

- ✅ 서버 상태 관리 최적
- ✅ 캐싱, 리페칭 자동화
- ✅ 낙관적 업데이트 지원

**사용 계획:**
- Phase 1 후반부터 도입 예정
- axios와 함께 사용하여 서버 상태 관리

### Zustand

**선택 이유:**

- ✅ 간단한 클라이언트 상태 관리
- ✅ Redux보다 가볍고 쉬움
- ✅ TypeScript 지원 우수

### axios

**선택 이유:**

- ✅ RESTful API 호출에 최적화
- ✅ 인터셉터로 공통 로직 처리 (인증, 에러 핸들링)
- ✅ TypeScript 지원 우수
- ✅ 요청/응답 변환 자동화
- ✅ API 엔드포인트 중앙 관리 (endpoints.ts)
- ✅ Query Key 중앙 관리 (queryKeys.ts)

**대안:**

- ❌ fetch: 인터셉터 없음, 보일러플레이트 많음
- ❌ Supabase SDK 직접 사용: DB 종속성 높음

**아키텍처:**

```
Frontend (Client Component)
    ↓ axios
API Routes (Next.js)
    ↓ Supabase SDK
Supabase (PostgreSQL)
```

## 🏗️ 아키텍처 패턴

### FSD (Feature-Sliced Design)

**레이어 구조:**

```
app/      → 라우팅, 페이지
widgets/  → 복합 컴포넌트 (Header 등)
features/ → 기능 단위 (auth, book 등)
entities/ → 비즈니스 엔티티 (book, user 등)
shared/   → 공통 UI, 설정, 유틸
```

**선택 이유:**

- ✅ 명확한 의존성 규칙
- ✅ 확장 가능한 구조
- ✅ 팀 협업에 유리
