> 작성일: 2025-01-03
> 목적: 계정 삭제 시 커뮤니티 데이터 보존 및 UX 일관성 유지

## 🎯 기본 원칙

### 1. 데이터 정책

**삭제되는 데이터**
- ✅ 독서 기록 (reading_records)
- ✅ **내가 올린 피드/활동 (activities)** - 완전 삭제
- ✅ AI 취향 분석 결과 (taste_analysis_results)
- ✅ 추천 도서 (book_recommendations)
- ✅ 알림 (notifications)
- ✅ 팔로우/팔로워 (follows)
- ✅ 내가 누른 좋아요 (likes)
- ✅ 프로필 정보 (profiles)
- ✅ 프로필 이미지 (Supabase Storage)

**보존되는 데이터 (커뮤니티 보호)**
- ✅ **다른 사람 글에 단 댓글** (comments)
  - `user_id`가 `NULL`로 설정됨
  - UI에서 "탈퇴한 사용자"로 표시
  - 대화 맥락 유지 및 다른 사용자 보호

**핵심 정책:**
- **주체 콘텐츠 (내 글)**: 완전 삭제 → 피드, 활동
- **반응 콘텐츠 (남의 글에 대한 댓글)**: 보존 → 댓글

### 2. UI 처리 원칙

**탈퇴한 사용자 식별:**
```typescript
const isDeletedUser = !comment.user || !comment.userId;
```

**표시 방법:**
- 닉네임: "탈퇴한 사용자" (회색 텍스트)
- 프로필 링크: **제거** (404 방지)
- 아바타: 기본 아이콘 (흐리게)

### 4. 알림에서 사용하기

```tsx
// src/features/notification/ui/NotificationItem.tsx
export const NotificationItem = ({ notification }: NotificationItemProps) => {
  const isDeletedUser = !notification.actor;

  // 탈퇴한 사용자의 알림은 메시지만 표시
  if (isDeletedUser) {
    return (
      <div className="notification">
        <UserAvatar isDeleted={true} />
        <span className="text-gray-500">
          탈퇴한 사용자가 {notification.message}
        </span>
      </div>
    );
  }

  return (
    <div className="notification">
      <UserAvatar
        photoUrl={notification.actor.photoUrl}
        nickname={notification.actor.nickname}
      />
      <UserLink
        userId={notification.actorId}
        username={notification.actor.username}
        nickname={notification.actor.nickname}
      />
      <span>{notification.message}</span>
    </div>
  );
};
```

## 🔧 DB 스키마

### 1. comments 테이블 (보존)

```sql
-- user_id를 nullable로 변경
ALTER TABLE comments
ALTER COLUMN user_id DROP NOT NULL;

-- ON DELETE CASCADE → ON DELETE SET NULL로 변경
ALTER TABLE comments
DROP CONSTRAINT IF EXISTS comments_user_id_fkey;

ALTER TABLE comments
ADD CONSTRAINT comments_user_id_fkey
FOREIGN KEY (user_id)
REFERENCES auth.users(id)
ON DELETE SET NULL;
```

**결과:**
- 사용자 계정 삭제 시 `user_id`가 `NULL`로 자동 설정
- 댓글 내용과 구조는 그대로 유지

### 2. activities 테이블 (삭제)

```sql
-- ON DELETE SET NULL → ON DELETE CASCADE로 변경
ALTER TABLE activities
DROP CONSTRAINT IF EXISTS activities_user_id_fkey;

ALTER TABLE activities
ADD CONSTRAINT activities_user_id_fkey
FOREIGN KEY (user_id)
REFERENCES auth.users(id)
ON DELETE CASCADE;
```

**결과:**
- 사용자 계정 삭제 시 내가 올린 피드/활동 완전 삭제
- 주체 콘텐츠는 삭제, 반응만 보존

## ✅ 체크리스트

댓글, 피드, 알림 등 사용자 정보를 표시하는 모든 곳에서:

- [ ] `user` 또는 `userId`가 `null`일 수 있는지 타입 확인
- [ ] `UserLink` 컴포넌트 사용 (프로필 링크 404 방지)
- [ ] `UserAvatar` 컴포넌트 사용 (탈퇴한 사용자 아이콘)
- [ ] "탈퇴한 사용자"는 수정/삭제 불가 처리
- [ ] Optional chaining (`user?.nickname`) 사용

## 📚 참고 사항

### Q. 왜 댓글은 보존하고 피드는 삭제하나요?

**주체 콘텐츠 vs 반응 콘텐츠**

1. **주체 콘텐츠 (내 글) → 삭제**
   - 피드, 활동 = 내가 주인공인 콘텐츠
   - 탈퇴 의사 = 내 흔적을 지우고 싶다는 의미
   - 예: "내가 책을 완독했어요" → 삭제

2. **반응 콘텐츠 (남의 글에 대한 반응) → 보존**
   - 댓글 = 다른 사람 글에 대한 의견
   - 삭제 시 대화 맥락 손상
   - 답글 단 다른 사용자 보호
   - 예: 다른 사람 피드에 "좋은 책이네요!" → 보존

3. **커뮤니티 가치 보존**
   - 도움이 되는 대화는 유지
   - 다른 사용자들의 경험 보호
   - GDPR 준수 (개인 식별 정보만 삭제)
