# 🛡️ Next.js 에러 처리 전략

> 작성일: 2025-12-04
> 목적: Next.js 환경에서 효과적인 에러 처리 방법과 적용 전략

---

## 📚 3가지 에러 처리 방법

| 방법                     | 파일/컴포넌트     | 용도                 | 적용 범위       | SSR 지원 |
| ------------------------ | ----------------- | -------------------- | --------------- | -------- |
| **1. `error.tsx`**       | Next.js 파일 규칙 | 라우트별 런타임 에러 | 해당 경로 전체  | ✅       |
| **2. `not-found.tsx`**   | Next.js 파일 규칙 | 404 에러 전용        | 해당 경로 404만 | ✅       |
| **3. `<ErrorBoundary>`** | React 컴포넌트    | 컴포넌트별 에러      | 감싼 컴포넌트만 | ❌       |

---

## 1️⃣ error.tsx - 라우트별 에러 처리

### 특징

- **자동 적용**: 해당 폴더의 `page.tsx`, `layout.tsx` 에러를 자동으로 캐치
- **SSR + CSR**: 서버/클라이언트 양쪽 에러 모두 처리
- **계층적 구조**: 하위에서 처리 못한 에러는 상위 `error.tsx`로 전파
- **필수 Client Component**: `'use client'` 지시어 필요

### 파일 구조

```
app/
  error.tsx              → 전체 앱의 에러
  books/
    error.tsx            → /books 경로의 에러만
    add/
      error.tsx          → /books/add 경로의 에러만
```

### 핵심 코드

```typescript
// app/error.tsx
'use client';

export default function Error({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  return (
    <div>
      <h2>오류가 발생했습니다</h2>
      <p>{error.message}</p>
      <button onClick={() => reset()}>다시 시도</button>
      <button onClick={() => window.location.href = '/'}>홈으로</button>
    </div>
  );
}
```

### Props

- `error.message`: 에러 메시지
- `error.digest`: 에러 고유 식별자
- `reset()`: 에러 상태 초기화 및 재렌더링

⚠️ **주의**: `layout.tsx`의 에러는 캐치하지 못함 (상위 레벨에 `error.tsx` 필요)

## 2️⃣ not-found.tsx - 404 에러 전용

### 특징

- **404 전용**: 존재하지 않는 리소스 처리
- **자동 + 수동**: URL 미매칭 또는 `notFound()` 함수 호출
- **Server Component 가능**: `'use client'` 불필요
- **SEO 친화적**: 올바른 404 상태 코드 반환

### 파일 구조

```
app/
  not-found.tsx          → 전역 404 페이지
  books/
    not-found.tsx        → /books 경로의 404 (우선순위 높음)
```

### 핵심 코드

```typescript
// app/not-found.tsx
export default function NotFound() {
  return (
    <div>
      <h1>404 - 페이지를 찾을 수 없습니다</h1>
      <a href='/'>홈으로 돌아가기</a>
    </div>
  );
}
```

### notFound() 함수 사용

```typescript
// app/books/[id]/page.tsx
import { notFound } from 'next/navigation';

export default async function BookDetailPage({ params }) {
  const book = await getBook(params.id);

  if (!book) {
    notFound(); // ← not-found.tsx 렌더링
  }

  return <div>{book.title}</div>;
}
```

## 3️⃣ ErrorBoundary - 컴포넌트별 에러 처리

### 특징

- **세밀한 제어**: 원하는 컴포넌트만 선택적으로 보호
- **부분 에러 처리**: 에러 발생해도 페이지 나머지는 정상 작동
- **CSR 전용**: 클라이언트 사이드 에러만 캐치
- **Class Component 필수**: React 18 현재 함수형 불가

### 핵심 코드

```typescript
// src/shared/ui/ErrorBoundary.tsx
'use client';

import { Component, type ReactNode } from 'react';

type ErrorBoundaryState = {
  hasError: boolean;
  error: Error | null;
};

export class ErrorBoundary extends Component<
  { children: ReactNode },
  ErrorBoundaryState
> {
  constructor(props) {
    super(props);
    this.state = { hasError: false, error: null };
  }

  static getDerivedStateFromError(error: Error) {
    return { hasError: true, error };
  }

  componentDidCatch(error: Error, errorInfo: React.ErrorInfo) {
    console.error('Error:', error, errorInfo);
  }

  render() {
    if (this.state.hasError) {
      return (
        <div>
          <h2>오류가 발생했습니다</h2>
          <button onClick={() => this.setState({ hasError: false })}>
            다시 시도
          </button>
        </div>
      );
    }
    return this.props.children;
  }
}
```

### 사용 예시

```typescript
<ErrorBoundary>
  <BookList /> {/* 여기서 에러 발생 시 ErrorBoundary가 캐치 */}
</ErrorBoundary>
```

## 🎯 실무 적용 전략

### 계층적 에러 처리 (권장)

```
app/
├── error.tsx                    ← 1단계: 전역 폴백
├── not-found.tsx               ← 404 전용
├── books/
│   ├── error.tsx              ← 2단계: 책 기능 전체 에러
│   ├── not-found.tsx          ← 책 관련 404
│   ├── page.tsx
│   │   └── <ErrorBoundary>    ← 3단계: 목록만 보호
│   │         <BookList />
│   │       </ErrorBoundary>
│   └── [id]/
│       └── page.tsx           ← notFound() 함수 사용
```

### 에러 처리 우선순위

```
1. notFound() 함수 호출
   → not-found.tsx 렌더링 (404 전용)

2. 런타임 에러 발생
   → <ErrorBoundary> (있으면 최우선)
   → error.tsx (현재 라우트)
   → error.tsx (상위 라우트, 계층적으로 올라감)
```

## 💡 언제 무엇을 사용할까?

### ✅ error.tsx

- 예상치 못한 모든 에러를 캐치하는 최후의 보루
- SSR 에러도 처리해야 할 때
- 라우트 전체를 보호해야 할 때

### ✅ not-found.tsx

- 존재하지 않는 리소스 요청
- 데이터베이스에 데이터가 없을 때
- SEO가 중요한 경우

### ✅ ErrorBoundary

- 특정 UI 영역만 보호하고 싶을 때
- 에러 발생해도 페이지 나머지는 정상 작동해야 할 때
- 세밀한 에러 UI 제어가 필요할 때

## ❌ 피해야 할 패턴

### 1. 중복 에러 처리

```typescript
// ❌ 나쁜 예: error.tsx가 있는데 ErrorBoundary 중복
<ErrorBoundary>
  <BooksPage /> {/* error.tsx가 이미 전체 처리 */}
</ErrorBoundary>

// ✅ 좋은 예: 특정 영역만 보호
<Header />
<ErrorBoundary>
  <BookList />
</ErrorBoundary>
<Footer />
```

### 2. 에러 무시

```typescript
// ❌ 나쁜 예
try {
  await fetchData();
} catch (error) {
  // 아무것도 안 함
}

// ✅ 좋은 예
try {
  await fetchData();
} catch (error) {
  console.error("데이터 로딩 실패:", error);
  setError("데이터를 불러올 수 없습니다.");
}
```
