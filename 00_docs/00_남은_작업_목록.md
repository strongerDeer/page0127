## ✅ 순위 변동 — 현재 상태 (2026-07-15 배포 QA로 갱신)

**DB**: 마이그레이션 적용 완료 (`20260713_create_ranking_snapshots.sql`). 배포 화면에 뱃지가
렌더되는 것 자체가 `get_book_ranking_with_delta` RPC 성공(폴백 아님)의 증거다.

**UI 렌더**: 랜딩 랭킹 3권 모두 `−`(순위 변동 없음)로 정상 렌더. `RankDeltaBadge`는
`has_history=false`면 아무것도 안 그리므로(28번 줄), `−`가 보인다는 것은
`has_history=true`(오늘 이전 스냅샷 존재) + `delta=0`이라는 뜻이다.

### ⚠️ "▲12"에 대한 정정 — 데이터가 3권뿐이라 순위가 고정이다

이전 판(07-14)에 "내일부터 진짜 `▲12`가 뜬다"고 적었으나 **부정확했다.**
`▲`는 순위가 실제로 뒤바뀌어야 뜬다. 지금 `most` 랭킹은 책 3권(완독 2/1/1명)이라
순위가 1·2·3으로 **고정**돼 `delta`가 영원히 0이다 → 매일 `−`만 뜬다.
이건 버그가 아니라 **데이터 부족의 자연스러운 결과**다(아래 "데이터가 거의 없다" 참조).
순위 변동이 살아나려면 코드가 아니라 **완독 데이터가 더 쌓여야** 한다.

### ❓ cron 자동 실행은 아직 미검증

- 현재 유일한 스냅샷 `2026-07-14`는 **수동 실행분**이다.
  `created_at`이 `2026-07-14 02:06 UTC`(= 11:06 KST)로, cron 예정 시각과 다르다.
- cron 스케줄은 `10 15 * * *` = **UTC 15:10 = KST 00:10**. `snapshot_date`는 DB(UTC)
  기준 `CURRENT_DATE`라, KST로는 하루 밀려 보인다
  (오늘 밤 KST 00:10 cron → `snapshot_date` = **07-15**).
- **`created_at`으로는 cron 실행 여부를 못 가린다.** 스냅샷 UPSERT가 `rank`/`count`만
  갱신하고 `created_at`은 안 건드려서(SQL 88-89번), cron이 같은 날짜 행을 덮어써도
  `created_at`은 최초값으로 남는다.
- **Hobby 플랜은 과거 런타임 로그를 거의 안 남긴다** → Vercel Logs로 어젯밤 실행을
  되짚는 건 사실상 불가.

| | cron 개수 | 최소 간격 | 정밀도 |

2026-07-14 실측:

**억지로 채우려 하지 말 것.** "오늘 0권 완독" 같은 문구는 오히려 역효과다(1-3 참조).
지금 코드는 데이터가 쌓이면 **자동으로 채워지도록** 짜여 있다. 필요한 건 코드가 아니라 책이다.

## 🎯 성공 기준 (재측정용)

리디자인 후 랜딩을 **같은 방식으로 재측정**한다. (측정 방법: playwright로 computed style 전수 조사)

| 지표 | 목표 | 현재 |
| --- | --- | --- |
| box-shadow | 2개 이하 | ✅ **0개** |
| 이모지 헤딩 | 0곳 | ✅ **0곳** |
| "당신" | 0회 | ✅ **0회** |
| "준비 중" 페이지 | 0개 | ✅ **0개** |
| 비로그인 진입 가능 콘텐츠 | 3개 이상 | ✅ **4개** |
| 화면의 날짜·시간 문자열 | 3개 이상 | ✅ **3종** |
| 순위 변동 표시 | 있음 | ⚠️ **뱃지 렌더는 정상**(`−`). `▲`는 책 3권 순위 고정이라 미출현 — 데이터 부족 |
| 폴드 안의 책 | 8권 이상 | ⚠️ 6권 — **데이터 부족** (DB에 4권뿐) |
| 랜딩 이미지 | 20개 이상 | ⚠️ 18개 — **데이터 부족** |

**디자인 목표는 전부 달성했다.** 남은 두 줄(폴드 안의 책·랜딩 이미지)은 디자인 문제가 아니라
**DB에 책이 4권뿐**이라서다. 책이 쌓이면 자동으로 채워진다.
**억지로 맞추려고 카드를 키우지 말 것.** 그러면 리디자인의 목적(교보·밀리처럼 밀도 있는 화면)과 정반대가 된다.

## 📎 참고: 확립된 원칙

Phase 0~5에서 정한 것. 새 화면을 만들 때 지킨다.

1. **그림자를 쓰지 않는다.** 입체는 `1px solid var(--line)`으로 만든다.
   그림자는 실제로 떠 있어야 하는 것(dialog·dropdown·popover·toast)에만.
2. **표지를 크롭하지 않는다.** 높이(또는 너비)만 고정하고 나머지는 원본 비율대로.
   판형의 불균일이 "진짜 책"의 증거다.
3. **letter-spacing을 건드리지 않는다.** Pretendard는 자간이 이미 최적화돼 있다.
4. **이모지를 헤딩·라벨에 쓰지 않는다.** lucide 단색 아이콘 또는 아이콘 없음.
5. **"당신"을 쓰지 않는다.** 주어를 생략하거나 `{닉네임}님`. AI 프롬프트에도 금지 지침이 있다.
6. **본문을 흐리게 칠하지 않는다.** 읽으라고 쓴 글은 `--text-strong` / `--text-body`.
7. **유채색은 직무를 하나씩 가진다.** 딥그린=완독, 테라코타=순위 상승, destructive=삭제.
8. **채워진 액센트 버튼은 화면당 최대 1개.**
9. **집계 화면에는 기준일을 박는다.** 날짜를 박는다는 건 갱신 책임을 진다는 선언이다.
10. **지어내지 않는다.** 업데이트 로그는 git 이력에서, 개인정보처리방침은 코드에서 확인한 것만.
11. **데이터가 없으면 우아하게 물러난다.** "오늘 0권 완독"은 살아 있다는 증거가 아니라
    **죽었다는 자백**이다. 숫자가 없으면 다른 문구로 물러나거나, 아예 그리지 않는다.
    없는 이력을 전부 `NEW`로 칠하지도 않는다(순위 변동의 `has_history`).
12. **없는 것을 "빈 결과"로 둔갑시키지 않는다.** `?? []` 폴백은 "함수가 없음"과
    "책이 없음"을 구분하지 못한다. 실패하면 `console.warn`으로 원인을 남긴다.
13. **환경변수에 `NEXT_PUBLIC_` 접두사를 함부로 붙이지 않는다.** 붙는 순간
    클라이언트 번들에 인라인된다. 서버 키는 접두사 없이.
14. **움직이는 것은 멈출 수 있어야 한다.** placeholder 회전·캐러셀은
    포커스 중과 `prefers-reduced-motion: reduce`에서 멈춘다.
15. **밀도는 면마다 다르게 — 발견 면은 감추고, 비교 면은 편다.** 같은 카드라도
    랜딩·랭킹(발견)은 소개·출판사·카테고리를 감추고, 전체목록·상세(비교)는 편다.
    단 **좋아요 같은 '액션'은 정보 필드가 아니라 밀도 분리의 대상이 아니다** — 감출 건
    비교 정보이지 상호작용이 아니다. 벤치마크(교보=판매)를 이식할 땐 우리 서비스의
    성격(취향)에 맞는지 먼저 따진다(저자를 뺄지 남길지가 갈린 지점).

## 🧭 착수 전 체크리스트 (매번 걸리는 것)

1. **"미착수"라고 적힌 작업도 실물을 먼저 확인한다.**
   1-2 상대 시간은 이미 구현돼 있었다. 문서가 코드보다 오래됐다.
2. **`git status`로 미커밋 WIP을 먼저 본다.**
   작업 중 사용자의 WIP(알라딘 키 서버 전용화)이 있었다.
3. **단정하기 전에 검산한다.**
   "service_role 키가 번들에 노출됐다"고 보고했으나 오탐이었다.
   Supabase 키는 전부 JWT라 앞부분(`eyJhbGci...`)이 anon 키와 같다.
   키 **전체**로 다시 검색해야 한다.
4. **화면의 신호가 무엇을 증명하고 무엇을 못 하는지 나눈다.** (2026-07-15)
   `−`(순위 변동 없음)가 뜨는 걸 보고 "cron이 돌고 있다"고 단정할 뻔했으나,
   `has_history`는 **오늘 이전 스냅샷 하나만** 있으면 참이라(수동분으로 충족)
   cron 자동실행과는 무관했다. `created_at`도 UPSERT가 안 건드려 실행 여부의 증거가
   못 된다. 화면·컬럼이 "있다"는 것과 "그 경로로 갱신됐다"는 것은 별개다.
5. **타임존을 먼저 계산한다.** (2026-07-15)
   cron `10 15 * * *`은 UTC라 KST 00:10이고, `snapshot_date`는 UTC `CURRENT_DATE`라
   KST 기준 하루 밀려 보인다. "왜 07-15가 아직 없지"의 답은 버그가 아니라 타임존이었다.
